## Technology responsibilities

      <li>File-based routes under <code>web/src/content/docs/</code></li>

    </ul>
  </article>

    <ul>
      <li>Primary sidebar generated from the shared page registry</li>
      <li>Reference pages for methodology and extension guides</li>
      <li>Accessible defaults retained under custom visual styling</li>
    </ul>
  </article>
  <article class="tool-card">
    <span class="eyebrow">MDX + ASTRO COMPONENTS</span>
    <h3>Composable content layer</h3>
    <p>MDX keeps long-form documentation readable while allowing reusable interface elements such as the tab bar, metric strip, forecast dashboard, analytics dashboard, and interactive Mermaid viewer.</p>
    <ul>
      <li>Documentation remains reviewable as text</li>
      <li>Complex visual components stay isolated</li>
      <li>New pages do not duplicate the site shell</li>

## Routing and artifact URLs

GitHub project Pages is hosted below a repository prefix rather than at the domain root. Every internal page and generated data artifact therefore passes through one build-time helper:

```ts
sitePath('forecast/')
sitePath('data/forecast.json')
```

The helper normalizes the Astro base to exactly one trailing slash. The final HTML receives explicit values such as:

```text
/media-experiment-ledger/forecast/
/media-experiment-ledger/data/forecast.json
```

<Aside type="note" title="Why build-time URLs?">
A data endpoint stored in a `data-*` attribute can be inspected in generated HTML and validated in CI. It also avoids relying on client-side assumptions about whether `BASE_URL` contains a trailing slash.
</Aside>

## Repository layout

```text
web/
  astro.config.mjs          Astro, Starlight, base path, sidebar
  navigation.mjs            primary-page registry
  package.json              pinned frontend dependencies
  public/data/              build inputs for analytics and forecasts
  src/
    components/             dashboards, tabs, metrics, Mermaid renderer
    content/docs/           MDX routes
    lib/sitePath.ts         base-safe internal URL builder
    styles/custom.css       visual system and responsive behavior
site/                       production output deployed to Pages
```

## Build and deployment

<MermaidDiagram
  title="Pages release path"
  caption="Source changes and formal experiment releases converge on the same reproducible build."
  diagram={`sequenceDiagram
    participant Dev as Source or Release event
    participant Action as GitHub Actions
    participant Python as Analytics + forecast tools
    participant Astro as Astro build
    participant Check as Site route validator
    participant Pages as GitHub Pages
    Dev->>Action: trigger workflow
    Action->>Python: produce canonical JSON artifacts
    Python-->>Action: analytics.json + forecast.json
    Action->>Astro: npm install and astro build
    Astro-->>Action: site/ static output
    Action->>Check: verify routes and base-safe data URLs
    Check-->>Action: pass or fail deployment
    Action->>Pages: upload and deploy site/
  `}
/>

The CI validator checks that every registered primary route exists in the production output, that Overview links include the repository base, and that both dashboard components expose valid base-prefixed JSON artifact URLs. This makes the routing contract part of the build rather than a manual browser check.
